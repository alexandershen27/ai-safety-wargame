# Forced-response tagging

> **Status:** Schema columns exist. No UI, no close-turn logic. **High priority for v0.2.**

## What it is

When you submit an action, you can tag another role with one of two intents:

- **forces-response** — the tagged role *must* address this in their next-turn
  action (or explicitly write "no response").
- **forbids-response** — the tagged role is barred from responding to this for
  N turns (variant; might cut from v1 of the feature).

Example flow:

> **Turn 4, USG (action):** "We ask @frontier-labs to put model-distillation
> safeguards on their open weights."
> *Tags `@frontier-labs` with forces-response.*

> **Vote phase:** players rate this 60% likely to pass (USG isn't actually
> compelling anything yet — they're just asking).

> **Resolve:** Reality writes "USG sends formal request. Frontier labs
> acknowledge receipt."

> **Turn 5, Frontier (forced action slot, marked):** "We add a known-watermark
> to our distillation outputs to satisfy the USG request without exposing weights."

That second action **is mandatory** for Frontier — they got an extra slot
this turn whether they wanted one or not. They can write "no response" if they
want to ignore it; that's still a discrete decision.

## Why it matters

This is the mechanic that makes deliberation *consequential*. Without it,
voting on actions is informational but not binding. Forced-response chains
turns into causal sequences: every action implicitly poses the question "who
else is involved, and what do they have to do about it?"

## Schema (already exists)

```sql
actions:
  is_forced            INTEGER NOT NULL DEFAULT 0     -- 1 if this slot was generated by a tag
  forced_by_action_id  TEXT                            -- pointer to the action that forced it

-- New table to add when implementing:
action_tags:
  id                   TEXT PRIMARY KEY
  action_id            TEXT NOT NULL REFERENCES actions(id)
  target_role_id       TEXT NOT NULL REFERENCES roles(id)
  kind                 TEXT NOT NULL                   -- 'forces-response' | 'forbids-response'
  UNIQUE (action_id, target_role_id)
```

## Implementation plan

### UI — Action draft (Discussion phase)
Below the textarea, a "Tag a role" row with the current world's roles as
clickable chips. Selecting a role pops a tiny menu: `forces response` /
`forbids response`. Tags appear as pills on the action card.

### UI — Vote phase
Render the forced-response chips on each action so voters can factor them into
their likelihood rating.

### UI — Discussion phase (next turn after a forced tag)
For each `action_tags` row pointing at a role I occupy, render an additional
draft card with a banner:
> **Forced response slot from Turn N.** USG asked you to address: "We ask
> frontier labs..." You can write your response or submit "no response".

That draft has `is_forced=true` and `forced_by_action_id=<tag's action id>`.

### Close-turn logic
In `src/app/api/worlds/[id]/advance/route.ts`, when transitioning to `CLOSED`:

1. For every `action_tags` row attached to a *resolved* action this turn:
   1. Find the target role.
   2. Insert a new action row in the *next* turn with:
      - `role_id = target_role_id`
      - `slot = next available slot for that role`
      - `is_forced = true`
      - `forced_by_action_id = <the tagging action's id>`
      - `draft_text = ""` (or a generated stub)
      - `author_player_id = (a seated player on that role, or NULL if unseated)`

   *Wrinkle:* if multiple players occupy the role, who owns the forced slot?
   First-come-first-served when they open the next turn, probably. Treat the
   row as "claimable" until someone starts drafting.

2. If a forced-response action exists but is never submitted, what happens?
   Two options:
   - **Auto-submit as "no response"** when phase advances. Lower-friction.
   - **Block phase advance until submitted.** Higher-friction but stricter.

   Default to auto-submit "no response" for now. Reality can manually edit.

## Open design questions

- Can Reality override a forced tag during Resolve? (e.g. "no, the USG can't
  actually compel this") — probably yes; Reality should always be able to
  reach into the schema.
- Can you tag your own role? (e.g. internal commitment) — probably no, UI
  should disable that.
- Does a forced action get its own vote phase? — yes, exactly like any
  other action. Tagging just guarantees the action exists.
- Multi-step chains: A tags B, B tags C in their forced response. Allowed.
  No special handling; the close-turn logic just runs again.
